# v0.1 task list: Runnable skeleton and health checks

## Goal

Deliver a reproducible local development environment in which the web app, core
API, knowledge service, PostgreSQL, and OpenSearch can be built, started, tested,
and observed through health checks. This release contains no document import,
search, authorization workflow, or AI answer generation.

## 1. Confirm the foundation

- [x] **V01-001 — Accept the core architecture decision.** Review ADR 0001,
  resolve remaining concerns, and set its status to `Accepted` or replace it.
- [ ] **V01-002 — Accept the retrieval security decision.** Review ADR 0002,
  resolve remaining concerns, and set its status to `Accepted` or replace it.
- [ ] **V01-003 — Define application names and local ports.** Record stable
  names, ports, base paths, and health endpoints for every service.
- [ ] **V01-004 — Select supported development tools.** Document required
  Docker, Node.js, Python, and JDK versions alongside the accepted Java, Spring
  Boot, and Gradle baseline.
- [ ] **V01-005 — Repair or initialize repository metadata.** Ensure the project
  is a valid Git repository and that generated files, secrets, local data, and
  private imports remain ignored.

## 2. Create the core API skeleton

- [ ] **V01-010 — Generate the Spring Boot application.** Create a Java 25,
  Spring Boot 4.1 application using the Gradle 9.6 Wrapper and Groovy DSL.
- [ ] **V01-011 — Configure the Java toolchain.** Pin Java 25 for compilation,
  tests, and execution without enabling preview features.
- [ ] **V01-012 — Establish module boundaries.** Create the initial package
  structure for application, domain, and infrastructure concerns without
  implementing future domain behavior.
- [ ] **V01-013 — Add health reporting.** Expose a minimal Spring Boot Actuator
  health endpoint without leaking configuration or environment details.
- [ ] **V01-014 — Configure tests.** Add Spock on the JUnit Platform, ArchUnit,
  and Testcontainers with versions verified against the selected baseline.
- [ ] **V01-015 — Add core API verification.** Cover application startup, the
  health endpoint, and the initial architecture boundary with automated tests.

## 3. Create the knowledge service skeleton

- [ ] **V01-020 — Select Python dependency management.** Record the supported
  Python version and choose one reproducible dependency and lock-file workflow.
- [ ] **V01-021 — Create the FastAPI application.** Add a minimal application,
  configuration model, production entry point, and development command.
- [ ] **V01-022 — Add health reporting.** Expose a small structured health
  endpoint that does not reveal secrets or host details.
- [ ] **V01-023 — Establish service boundaries.** Reserve clear modules for
  extraction, indexing, retrieval, and answer generation without implementing
  them in v0.1.
- [ ] **V01-024 — Configure quality checks.** Add formatting, linting, static
  analysis, and pytest using repository-owned configuration.
- [ ] **V01-025 — Add knowledge service verification.** Test application startup
  and the health response without requiring external AI services.

## 4. Create the web skeleton

- [ ] **V01-030 — Select the frontend baseline.** Record the supported Node.js,
  package-manager, TypeScript, and Angular versions and commit the lock file.
- [ ] **V01-031 — Generate the Angular application.** Create the application
  with routing, strict TypeScript settings, and production build support.
- [ ] **V01-032 — Add runtime configuration.** Define API base URLs without
  embedding secrets or machine-specific values in source code.
- [ ] **V01-033 — Create the system status page.** Display whether the core API
  and knowledge service are reachable and handle loading and failure states.
- [ ] **V01-034 — Configure frontend quality checks.** Add formatting, linting,
  unit tests, and a production build.
- [ ] **V01-035 — Add web verification.** Test status presentation for healthy,
  unavailable, and malformed service responses.

## 5. Provide local infrastructure

- [ ] **V01-040 — Add PostgreSQL to Compose.** Pin an image version, configure a
  health check and named development volume, and use non-production credentials.
- [ ] **V01-041 — Add OpenSearch to Compose.** Pin a compatible image version,
  configure single-node development mode, a health check, resource limits, and
  a named development volume.
- [ ] **V01-042 — Containerize application services.** Add reproducible,
  non-root development/runtime images for the core API and knowledge service.
- [ ] **V01-043 — Add the web runtime.** Serve the production Angular build in a
  local container and configure routing to the backend services.
- [ ] **V01-044 — Define Compose dependencies.** Start dependants only after the
  required infrastructure is healthy and provide useful failure diagnostics.
- [ ] **V01-045 — Add environment templates.** Commit `.env.example` containing
  safe development defaults and document every setting; never commit `.env`.
- [ ] **V01-046 — Document data lifecycle.** Provide explicit commands for normal
  shutdown and for intentionally removing local development volumes.

## 6. Define contracts and observability

- [ ] **V01-050 — Define health contracts.** Add versioned OpenAPI definitions
  for the externally consumed health/status responses.
- [ ] **V01-051 — Standardize error responses.** Define the minimal machine-
  readable error shape used across service boundaries in v0.1.
- [ ] **V01-052 — Configure safe logging.** Use structured, readable logs with
  service identity and correlation support, excluding secrets and request bodies.
- [ ] **V01-053 — Verify dependency health behavior.** Distinguish application
  liveness from readiness where an unavailable dependency affects readiness.

## 7. Create portable developer workflows

- [ ] **V01-060 — Add build entry points.** Provide documented commands that
  build all applications without depending on a CI provider.
- [ ] **V01-061 — Add test entry points.** Provide commands for fast unit checks
  and for integration tests that require containers.
- [ ] **V01-062 — Add a repository-wide verify entry point.** Run formatting,
  linting, static analysis, architecture tests, unit tests, and production builds.
- [ ] **V01-063 — Add lifecycle entry points.** Provide simple `up`, `status`,
  `logs`, and `down` workflows for the complete local environment.
- [ ] **V01-064 — Support Windows and CI usage.** Ensure the documented entry
  points work in the primary Windows environment and in the selected CI runner.

## 8. Add continuous integration

- [ ] **V01-070 — Add the CI workflow.** Invoke repository-owned verification
  commands for pull requests and the default branch.
- [ ] **V01-071 — Configure safe caching.** Cache immutable dependency artifacts
  without caching secrets, private documents, or runtime data.
- [ ] **V01-072 — Run integration verification.** Start required containers,
  exercise health paths, and retain useful diagnostics on failure.
- [ ] **V01-073 — Add dependency update visibility.** Configure a maintainable
  mechanism for proposed dependency updates without automatic merging.

## 9. Document and accept v0.1

- [ ] **V01-080 — Write the quick start.** Document prerequisites, initial setup,
  startup, verification, logs, shutdown, and troubleshooting from a clean clone.
- [ ] **V01-081 — Document service topology.** Record services, ports, health
  endpoints, dependencies, configuration sources, and data volumes.
- [ ] **V01-082 — Verify license and data safety.** Confirm that the release
  contains no private imports, secrets, or restricted source material.
- [ ] **V01-083 — Perform a clean-environment rehearsal.** Follow only the public
  documentation on a clean checkout and record the exact verification result.
- [ ] **V01-084 — Tag the release.** Create `v0.1.0` only after every acceptance
  criterion below is satisfied.

## Acceptance criteria

- [ ] A new contributor can prepare and start the project using only the
  repository documentation.
- [ ] One documented workflow starts all five components: web, core API,
  knowledge service, PostgreSQL, and OpenSearch.
- [ ] Every component exposes or participates in a meaningful health check.
- [ ] The web status page represents healthy and unavailable backend services.
- [ ] Repository-wide verification passes locally and in CI.
- [ ] Builds use Java 25 without preview features and the committed Gradle Wrapper.
- [ ] No external AI provider, private document, or secret is required.
- [ ] Local persistent data can be retained during normal restarts and removed
  through a separate, explicitly destructive command.

## Explicitly out of scope

- document upload, extraction, chunking, and indexing
- user, campaign, membership, and permission workflows
- full-text, vector, or hybrid search
- source-backed AI answers and model-provider integration
- production deployment, production secrets, and high-availability operation
